# Phase 5: Memory Management - Research

**Researched:** 2026-03-10
**Domain:** Filesystem memory browser with markdown editing in Next.js dashboard
**Confidence:** HIGH

## Summary

Phase 5 adds a Memory tab to the agent slide-out panel that lets users browse and edit markdown files from an agent's workspace directory. The implementation is entirely frontend + Next.js API route work, building on well-established patterns already in the codebase. The core challenge is a filesystem tree scanner API that recursively reads an agent's workspace for `.md` files, a file read/write API with path traversal protection, and a two-column UI (file tree + editor/viewer) inside the existing slide-out panel.

The codebase already has every building block needed: `resolveWithin()` for path safety, `requireRole()` for RBAC, `MarkdownRenderer` for preview, `useToast()` for notifications, `AgentSlideOut` for the tab container, and `DiscoveredAgent.workspacePath` for the filesystem root. The existing `/api/agents/[id]/soul/route.ts` demonstrates the exact read/write pattern to extend. No new dependencies are required.

**Primary recommendation:** Build two new API routes (`/api/agents/[id]/files` for tree listing, `/api/agents/[id]/files/[...path]` for read/write) and three new UI components (MemoryTab, FileTree, FileEditor) following existing codebase patterns exactly.

## Architecture Patterns

### Recommended API Structure
```
src/app/api/agents/[id]/
  files/
    route.ts          # GET: recursive file tree listing
    [...path]/
      route.ts        # GET: read file content, PUT: write file content
```

**Why two routes, not one:**
- `GET /api/agents/{id}/files` returns the full tree structure (names, types, timestamps)
- `GET /api/agents/{id}/files/{path}` returns a single file's content
- `PUT /api/agents/{id}/files/{path}` writes a single file's content
- This mirrors REST conventions and keeps tree scanning separate from file I/O

### Recommended Component Structure
```
src/components/agents/
  agent-memory-tab.tsx          # Main Memory tab (two-column layout)
  memory-file-tree.tsx          # Left panel: recursive file tree
  memory-file-editor.tsx        # Right panel: viewer/editor with toggle
```

### Pattern 1: Recursive Filesystem Tree Scanner
**What:** API endpoint that recursively scans an agent's workspace for .md files and returns a tree structure
**When to use:** GET /api/agents/{id}/files
**Key considerations:**
- Use `DiscoveredAgent.workspacePath` as the root (already available via healthMonitor.getSnapshot())
- Filter to only `.md` files and directories that contain `.md` files (recursively)
- Include `fs.statSync().mtimeMs` for last-modified timestamps
- Use `resolveWithin()` to prevent path traversal on every path operation
- Skip non-UTF8 files silently (catch decode errors, return error marker)
- Synchronous reads are acceptable (< 10 agents, < 100 files per agent, server-side)

**Example tree response:**
```typescript
// Source: Derived from existing MemoryFile type in shared.ts
interface MemoryFileNode {
  path: string          // relative path from workspace root, e.g. "SOUL.md" or "memory/MEMORY.md"
  name: string          // filename, e.g. "SOUL.md"
  type: 'file' | 'directory'
  modified?: number     // Unix timestamp (ms)
  readOnly?: boolean    // true for HEARTBEAT.md, SESSION-STATE.md
  children?: MemoryFileNode[]
}
```

### Anti-Patterns to Avoid
- **Caching the file tree in Zustand:** Files change frequently (agents write to memory). Always fetch fresh from the API when the tab opens or file is selected.
- **Using the existing `/api/agents/[id]/memory` route:** This is DB-based working memory, completely different from filesystem memory files. Create new routes at `/api/agents/[id]/files/`.
- **Reading files client-side:** All filesystem access must go through API routes. Browser never talks directly to the filesystem.
- **Allowing edits to HEARTBEAT.md / SESSION-STATE.md:** These are auto-generated. Enforce read-only both server-side (reject PUT) and client-side (hide Edit button).

## Common Pitfalls

### Pitfall 1: Path Traversal via Relative Paths
**What goes wrong:** User-supplied file paths like `../../etc/passwd` escape the agent workspace
**Why it happens:** API routes accept arbitrary path strings from the client
**How to avoid:** Every path operation MUST use `resolveWithin(workspacePath, userPath)`. Wrap in try/catch -- resolveWithin throws on escape attempts. Return 400 on path escape.
**Warning signs:** Any `path.join()` or `path.resolve()` that doesn't go through `resolveWithin()`

### Pitfall 2: Conflicting Route with Existing Memory API
**What goes wrong:** New filesystem memory routes conflict with existing `/api/agents/[id]/memory/route.ts` (DB-based working memory)
**Why it happens:** Next.js App Router route segments can collide
**How to avoid:** Use a different route segment: `/api/agents/[id]/files/` instead of `/api/agents/[id]/memory/`. The existing memory route stays untouched for backward compatibility.
**Warning signs:** 404s or unexpected responses when hitting memory endpoints

### Pitfall 3: Discovered Agent Not Found
**What goes wrong:** API returns 404 because healthMonitor hasn't run yet or agent ID doesn't match
**Why it happens:** On first load, healthMonitor may not have completed initial scan. Agent IDs are directory names (e.g., "stefany"), case-sensitive.
**How to avoid:** Start healthMonitor on route access (already done in discover/route.ts). Use exact ID matching. Return clear "Agent not found or not yet discovered" error.
**Warning signs:** Intermittent 404s on first page load

### Pitfall 4: Non-UTF8 File Content
**What goes wrong:** `readFileSync(path, 'utf-8')` throws or corrupts binary content
**Why it happens:** Some .md files might have unusual encodings or the scan picks up files that aren't truly text
**How to avoid:** Wrap file reads in try/catch. If readFileSync fails, return `{ error: 'File not readable', encoding: 'unknown' }` in the API response. UI shows "This file cannot be displayed" message.
**Warning signs:** Unhandled exceptions in API routes

### Pitfall 5: Race Condition on Save
**What goes wrong:** User edits a file while the agent is actively writing to it, causing data loss
**Why it happens:** Agent and dashboard both write to the same files
**How to avoid:** This is acknowledged in the user's design -- show yellow "Agent is running" warning banner. Detect running state from `health.dimensions.process.level !== 'red'` (same check used in overview tab). No locking mechanism -- save anyway with warning.
**Warning signs:** Silent data overwrites

### Pitfall 6: Slide-Out Panel Width Constraint
**What goes wrong:** Two-column layout (200px tree + editor) is cramped in a 400px panel
**Why it happens:** Current slide-out is `w-full md:w-[400px]`
**How to avoid:** Consider widening the panel when Memory tab is active (e.g., `md:w-[600px]`). The CONTEXT.md mentions "Slide-out is ~600px wide, works well." This means the panel width should be increased for the memory tab.
**Warning signs:** Textarea is too narrow to edit comfortably

## Code Examples

### File Tree Scanner (API Server-Side)
```typescript
// Source: Pattern derived from agent-discovery.ts + paths.ts
import fs from 'node:fs'
import path from 'node:path'
import { resolveWithin } from '@/lib/paths'

interface MemoryFileNode {
  path: string
  name: string
  type: 'file' | 'directory'
  modified?: number
  readOnly?: boolean
  children?: MemoryFileNode[]
}

const READ_ONLY_FILES = new Set(['HEARTBEAT.md', 'SESSION-STATE.md'])

function scanWorkspace(workspacePath: string): MemoryFileNode[] {
  const results: MemoryFileNode[] = []

  function scanDir(dirPath: string, relativeTo: string): MemoryFileNode[] {
    const entries = fs.readdirSync(dirPath, { withFileTypes: true })
    const nodes: MemoryFileNode[] = []

    for (const entry of entries) {
      if (entry.name.startsWith('.')) continue

      const fullPath = path.join(dirPath, entry.name)
      const relPath = path.relative(relativeTo, fullPath)

      if (entry.isFile() && entry.name.endsWith('.md')) {
        const stat = fs.statSync(fullPath)
        nodes.push({
          path: relPath,
          name: entry.name,
          type: 'file',
          modified: stat.mtimeMs,
          readOnly: READ_ONLY_FILES.has(entry.name),
        })
      } else if (entry.isDirectory()) {
        const children = scanDir(fullPath, relativeTo)
        // Only include directories that contain .md files (at any depth)
        if (children.length > 0) {
          nodes.push({
            path: relPath,
            name: entry.name,
            type: 'directory',
            children,
          })
        }
      }
    }

    // Sort: directories first, then files, alphabetical within each group
    return nodes.sort((a, b) => {
      if (a.type !== b.type) return a.type === 'directory' ? -1 : 1
      return a.name.localeCompare(b.name)
    })
  }

  return scanDir(workspacePath, workspacePath)
}
```

### File Read/Write (API Server-Side)
```typescript
// Source: Pattern from soul/route.ts
import { resolveWithin } from '@/lib/paths'

// Read
const safePath = resolveWithin(workspacePath, filePath)
if (!fs.existsSync(safePath)) {
  return NextResponse.json({ error: 'File not found' }, { status: 404 })
}
const content = fs.readFileSync(safePath, 'utf-8')
const stat = fs.statSync(safePath)

return NextResponse.json({
  path: filePath,
  content,
  modified: stat.mtimeMs,
  readOnly: READ_ONLY_FILES.has(path.basename(filePath)),
})

// Write
const safePath = resolveWithin(workspacePath, filePath)
if (READ_ONLY_FILES.has(path.basename(filePath))) {
  return NextResponse.json({ error: 'File is read-only' }, { status: 403 })
}
fs.writeFileSync(safePath, content, 'utf-8')
```

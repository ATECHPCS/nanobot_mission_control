  <behavior>
    - GET /api/token-stats?range=week returns UnifiedTokenStats with summary, byAgent, byModel, timeline
    - summary.totalInputTokens and summary.totalOutputTokens aggregate from both claude_sessions and token_usage tables
    - summary.totalSessions counts from claude_sessions + nanobot_sessions (message counts, not token counts for nanobot)
    - summary.mostActiveAgent is the agent/project_slug with most total tokens
    - byAgent groups Claude Code tokens by project_slug and nanobot sessions by agent_id (message counts only for nanobot)
    - byModel groups by model field from both claude_sessions and token_usage, splitting input/output
    - timeline produces date-bucketed data points for the selected range
    - range parameter: "today" = last 24h, "week" = 7 days (default), "month" = 30 days, "year" = 365 days
  </behavior>
  <action>
    **1. Create `src/app/api/token-stats/route.ts`:**
    - `GET` handler with `requireRole('viewer')`
    - Accept query param `range` (today|week|month|year, default: week)
    - Calculate cutoff datetime based on range
    - **Query 1 -- Claude Code session tokens:**
      ```sql
      SELECT project_slug, model, SUM(input_tokens) as input_tokens, SUM(output_tokens) as output_tokens,
             COUNT(*) as session_count, MIN(first_message_at) as first_at, MAX(last_message_at) as last_at
      FROM claude_sessions
      WHERE last_message_at >= ?
      GROUP BY project_slug, model
      ```
    - **Query 2 -- MC API token_usage:**
      ```sql
      SELECT model, SUM(input_tokens) as input_tokens, SUM(output_tokens) as output_tokens,
             COUNT(*) as request_count
      FROM token_usage
      WHERE created_at >= ?
      GROUP BY model
      ```
    - **Query 3 -- Nanobot session counts (message counts, NOT token counts):**
      ```sql
      SELECT agent_id, SUM(message_count) as total_messages, COUNT(*) as session_count
      FROM nanobot_sessions
      WHERE last_message_at >= ?
      GROUP BY agent_id
      ```
    - **Query 4 -- Timeline (date bucketing):**
      For Claude Code sessions: group by date(last_message_at), sum input/output tokens.
      For token_usage: group by date(created_at, 'unixepoch'), sum input/output tokens.
      Merge both into a single timeline array sorted by date.
    - **Aggregate into response shape:**
      ```typescript
      interface UnifiedTokenStats {
        summary: {
          totalInputTokens: number
          totalOutputTokens: number
          totalSessions: number      // claude_sessions count + nanobot_sessions count
          totalMessages: number      // nanobot session message sum
          mostActiveAgent: string    // agent/project with most tokens (or messages for nanobot)
          avgTokensPerSession: number
        }
        byAgent: Array<{
          agent: string              // agent_id for nanobot, project_slug for claude code
          source: 'nanobot' | 'claude-code'
          inputTokens: number
          outputTokens: number
          sessionCount: number
          messageCount: number       // only for nanobot agents
        }>
        byModel: Array<{
          model: string
          inputTokens: number
          outputTokens: number
        }>
        timeline: Array<{
          date: string               // YYYY-MM-DD
          inputTokens: number
          outputTokens: number
        }>
        range: string
      }
      ```
    - Before querying, call `syncClaudeSessions()` to refresh Claude Code session data (it's fast -- just re-scans JSONL files and upserts).
    - Handle empty tables gracefully -- return zeroed summary with empty arrays.

    **2. Create `src/lib/__tests__/token-aggregation.test.ts`:**
    - Test the SQL queries by setting up an in-memory SQLite database with test data
    - Test: Claude Code sessions contribute to byModel and byAgent with correct input/output split
    - Test: token_usage rows contribute to byModel
    - Test: nanobot_sessions contribute message counts (NOT token counts) to byAgent
    - Test: timeline merges both sources into date-bucketed array
    - Test: range filtering (today vs week vs month) produces correct cutoffs
    - Test: empty tables return zeroed stats
    - Use `better-sqlite3` directly for test DB setup (follow existing test patterns)

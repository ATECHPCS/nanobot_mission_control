    **1. Extend Zustand store (src/store/index.ts):**

    Add to the MissionControlStore interface:
    ```typescript
    // Lifecycle Operations (Phase 3)
    lifecycleOperations: Map<string, LifecycleOperation>  // agentId -> current in-progress operation
    lifecycleHistory: LifecycleOperation[]  // last 10 per agent, all agents combined, newest first
    setLifecycleOperation: (agentId: string, op: LifecycleOperation | null) => void
    addLifecycleHistory: (op: LifecycleOperation) => void
    getAgentLifecycleHistory: (agentId: string) => LifecycleOperation[]
    isAgentLocked: (agentId: string) => boolean
    ```

    Import `LifecycleOperation` from `@/types/agent-health`.

    Implementation notes:
    - `lifecycleOperations`: initialized as `new Map()`. `setLifecycleOperation` sets or deletes (if op is null) an entry. Trigger a re-render by creating a new Map instance: `new Map(state.lifecycleOperations)`.
    - `lifecycleHistory`: capped at 100 entries total. `addLifecycleHistory` prepends to array, slices to 100. Also calls `setLifecycleOperation(op.agentId, null)` when op.status is 'success' or 'error' (clears the in-progress operation).
    - `getAgentLifecycleHistory(agentId)`: filters lifecycleHistory by agentId, returns last 10.
    - `isAgentLocked(agentId)`: returns `lifecycleOperations.has(agentId)`.

    IMPORTANT: Since Zustand's `Map` objects don't trigger re-renders automatically, the `setLifecycleOperation` action must clone the Map: `set({ lifecycleOperations: new Map(state.lifecycleOperations) })` after mutation.

    **2. Handle SSE lifecycle events (src/lib/use-server-events.ts):**

    Find the event dispatch switch/if-chain in the existing `useServerEvents` hook. Add a case for `'agent.lifecycle'`:
    ```typescript
    case 'agent.lifecycle': {
      const { id, action, status, username, timestamp, error } = event.data
      const op: LifecycleOperation = { agentId: id, action, status, timestamp, username, error }

      if (status === 'pending') {
        store.setLifecycleOperation(id, op)
      } else {
        // 'success' or 'error' -- add to history and clear active operation
        store.addLifecycleHistory(op)
      }
      break
    }
    ```

    Also: when `status === 'success'` for a start or restart action, trigger a health refresh (the agent's health snapshot will be stale). Call the existing refresh mechanism if available, or set a flag to trigger re-fetch on next poll cycle.

    **2. Create Lifecycle tab (src/components/agents/agent-lifecycle-tab.tsx):**

    'use client' component. Props: `{ snapshot: AgentHealthSnapshot }`.

    Sections (top to bottom):
    a) **Launch command display:**
       - If `snapshot.agent.launchScript` is non-empty: show the script path in a monospace code block, labeled "Launch script"
       - If launchScript is empty (root agent): show `nanobot gateway --port {gatewayPort}`, labeled "Inferred from config"

    b) **No gateway port state:**
       - If `snapshot.agent.gatewayPort` is 0 or falsy: show a muted message "No gateway port configured. Lifecycle controls require a configured gateway port." and return early (no buttons, no history).

    c) **Action buttons (context-aware per user decision):**
       - Determine agent alive status from `snapshot.health.dimensions.process.level` -- 'green' means alive
       - Use `useMissionControl()` to check `isAgentLocked(snapshot.id)` for disabling buttons during operations
       - Get `currentUser` from store for username in API calls
       - If agent is alive (process.level === 'green'):
         - Show "Stop" button (text-destructive styling) -- clicking opens ConfirmModal
         - Show "Restart" button (text-warning/amber styling) -- clicking opens ConfirmModal
       - If agent is dead/yellow:
         - Show "Start" button (text-primary styling) -- clicking calls API directly (no modal per user decision: "starting is non-destructive")
       - If locked (operation in progress): show the current operation status text ("Starting...", "Stopping...", "Restarting...") instead of buttons. Use a small spinner icon next to the text.
       - All buttons: `px-3 py-1.5 text-xs rounded-md font-medium` styling

    d) **Force Kill button:**
       - Track a local state `forceKillAvailable` with a 10-second timer
       - When a stop operation is pending (from store), start a 10s setTimeout. When it fires, set `forceKillAvailable = true`.
       - When forceKillAvailable is true and stop is still pending: show "Force Kill" button (red/destructive styling)
       - Force Kill calls `POST /api/agents/{id}/force-stop` directly (no modal -- the user already confirmed the stop)
       - Reset forceKillAvailable when operation completes

    e) **Operation history:**
       - Get history via `useMissionControl().getAgentLifecycleHistory(snapshot.id)` (returns last 10, but show at most the last 5 or 10 per decision)
       - Render as a list: each entry shows action icon/label, status badge (success=green, error=red), timestamp (formatted), username
       - Format: "Stopped by alice at 2:30pm" or "Start failed at 2:31pm"
       - If error present, show error message as muted text below the entry (this is the inline error display per user decision)
       - If history is empty: show "No operations yet" in muted text
       - Use smart error hints: if error contains "address already in use" or "EADDRINUSE", show "Port {port} is already in use". If error contains "not found", show "nanobot binary not found in PATH".

    f) **API call functions:**
       - `handleStart()`: POST `/api/agents/${snapshot.id}/start`. On error response, show toast via existing toast system.
       - `handleStop()`: POST `/api/agents/${snapshot.id}/stop`. On error response, show toast.
       - `handleRestart()`: POST `/api/agents/${snapshot.id}/stop`, wait for completion (SSE event), then POST start. Actually simpler: just POST stop and the server handles the full restart cycle. Actually per the API design, there's no restart endpoint -- restart is stop + start. The simplest approach: POST stop, watch for success via SSE, then POST start. Use the lifecycleOperations store state to track progress.
       - Alternative simpler approach: call stop, and in the SSE handler for stop success, automatically trigger start if the original action was 'restart'. Track `pendingRestart` in local component state. When SSE stop.success arrives and pendingRestart is true, call start.
       - `handleForceStop()`: POST `/api/agents/${snapshot.id}/force-stop`

    g) **Confirmation modal integration:**
       - Local state: `confirmAction: 'stop' | 'restart' | null`
       - When Stop clicked: set confirmAction to 'stop'
       - When Restart clicked: set confirmAction to 'restart'
       - Modal title: "Stop {agentName}?" or "Restart {agentName}?"
       - Modal message: "This will interrupt any active sessions. The agent will stop processing messages until restarted." (for stop) or "This will briefly interrupt active sessions while the agent restarts." (for restart)
       - Modal confirm button label: "Stop Agent" or "Restart Agent"
       - Modal destructive: true (both stop and restart are destructive actions per user decision)
       - On confirm: close modal immediately (set confirmAction to null), call handleStop or handleRestart
